## 2. 렌더링 파이프라인

**Q. 렌더링 파이프라인부터 설명해줘. 대신 길게 설명하지 말고 하나씩 끊어서.**

브라우저가 HTML을 받으면 위에서 아래로 한 줄씩 읽으면서 DOM 트리를 만들어.

```
Document
└── html
    ├── head
    └── body
        └── div#root
            └── h1 ("코드도보통지")
```

HTML은 문자열, DOM은 살아있는 객체야.

---

**Q. html 문서가 생성되면 그냥 브라우저에서 렌더링을 하는 게 아니라 메모리에 DOM 트리를 저장? 한다고 이해하면 될까?**

맞아, 정확해. 브라우저가 HTML을 받으면 그냥 바로 화면에 그리는 게 아니라 먼저 메모리에 DOM 트리를 구성해.

```
HTML = 설계도
DOM  = 메모리에 올라간 실물
```

---

**Q. 저연차 때 "노드"라는 말을 이해하는 데까지 1년여의 세월을 허비했지 ㅋㅋㅋㅋ**

노드 = 이름표 붙은 상자. 상자 하나가 노드야. 텍스트도 상자, 태그도 상자.

"트리"는 이 상자들이 부모-자식으로 연결된 구조. "가지"는 그림으로 그리면 나무처럼 생겨서 붙인 이름인데 설명으로 쓰면 최악이지 ㅋㅋ

---

**Q. 나무를 거꾸로 뒤집어서 놓으니 보이더라고 뿌리(root)가 있고 각각 뻗어나가는 가지들**

ㅋㅋㅋㅋ 맞아 그거야!! 거꾸로 뒤집힌 나무. 그래서 `root`가 뿌리고, `children`이 자식 가지들이고, `parent`가 위 가지야. 용어들이 전부 그 나무 비유에서 온 거거든.

---

**Q. 요소 - 속성 - 텍스트 - 가상요소 각각의 모든 것들이 노드!**

오 정확해! 한 번에 정리됐네 ㅋㅋ

```js
node.nodeType === 1  // 요소 노드
node.nodeType === 2  // 속성 노드
node.nodeType === 3  // 텍스트 노드
```

브라우저 입장에서는 전부 "트리 위의 노드"로 동일하게 취급해.

---

**Q. cssom? css object model 이야?**

응 맞아! CSSOM — CSS Object Model.

DOM이 HTML을 메모리에 올린 것처럼, CSSOM은 CSS를 메모리에 올린 거야. DOM 트리랑 동시에 만들어져. 둘 다 완성되면 다음 단계로.

---

**Q. 객체?**

맞는 방향이야! DOM도 객체 트리, CSSOM도 객체 트리. 그럼 이 둘을 합쳐야 화면에 그릴 수 있겠지?

```
DOM 트리  +  CSSOM 트리  →  Render Tree
```

---

**Q. 렌더 트리는 처음 들어본다 ㅋ**

당연해, 브라우저 내부에서 조용히 만들어지는 거라 눈에 안 보이거든.

```
DOM 트리    →  "뭐가 있는가" (구조)
CSSOM 트리  →  "어떻게 생겼는가" (스타일)
Render 트리 →  "실제로 화면에 그릴 것만" (둘의 교집합)
```

`display: none`인 요소는 DOM에는 있지만 Render 트리에서 빠져.

---

**Q. display?**

맞아!

```
display: none     → Render 트리에서 아예 제거 (없는 취급)
visibility: hidden → Render 트리에 있음 (자리는 차지, 눈에만 안 보임)

visibility: hidden  →  투명인간 (자리는 차지)
display: none       →  아예 퇴근 (자리 자체가 없음)
```

---

**Q. 렌더링을 진행하겠지?**

맞아! 근데 렌더링 전에 한 단계가 더 있어. 렌더 트리엔 아직 위치와 크기 정보가 없어.

**Layout (= Reflow)** — 모든 노드의 정확한 위치와 크기를 계산하는 단계.

하나가 바뀌면 연쇄적으로 재계산이 일어나. 비용이 많이 들어.

---

**Q. css에 transition을 입힐 때 퍼포먼스 어쩌고 할 때 reflow와 repaint 이야기를 들었던 거 같아**

오 맞아! 이제 다음 단계가 바로 **Paint (= Repaint)**거든.

```
Layout  →  위치/크기 계산     Reflow  →  Layout부터 다시  💸💸💸
Paint   →  색상/배경 채우기   Repaint →  Paint만 다시     💸
```

---

**Q. 위치 크기 계산이 필요 없으니까?**

맞아! 근데 거기서 한 발 더 나아가.

`transform`은 Layout도, Paint도 건드리지 않아. 아예 별도 레이어에서 처리돼 — **Composite**.

```css
transition: width 0.3s       /* Reflow 유발 😩 */
transition: background 0.3s  /* Repaint 유발 🙂 */
transition: transform 0.3s   /* Composite만 😍 */
```

`transform`은 GPU가 처리. 포토샵 레이어처럼 자기 레이어만 움직여.

---

**Q. 퍼포먼스 관점에서(lighthouse 활용) layout, paint를 면밀히 고려하면 되는 건가?**

```
LCP  →  가장 큰 요소가 화면에 그려지는 시간  →  Paint 단계
CLS  →  화면 덜컹거림                        →  Layout 단계
INP  →  클릭 후 화면 반응 속도               →  Layout + Paint 전체
```

Lighthouse는 결과 수치를 알려주고, 파이프라인 이해가 있어야 원인을 찾을 수 있어.

---

**Q. skeleton을 활용하는 방법이 layout 때문인거야?**

skeleton은 Layout보다는 UX 관점이 더 커. CLS를 잡는 패턴이야:

```
skeleton이 자리 미리 차지 → Layout 미리 계산
데이터 도착 → skeleton 자리에 콘텐츠 교체
→ Layout 변화 없음 → CLS 없음 ✅
```

---

**Q. 파이프라인을 내가 이해했는 지 한 번 봐줘**
```
1. 서버로부터 html 전달 받음
2. 전달받은 html을 분석해서 dom tree 생성 후 메모리에 캐시
3. css 확인
4. cssom 메모리에 캐싱
5. layout을 통해 위치/크기 렌더링
6. paint를 통해 색상 렌더링
7. 레이어 합성
```

거의 완벽한데 두 군데만 보완:
- 3~4번 사이에 Render 트리 단계 빠짐
- "캐시"보다 "메모리에 올린다"가 더 정확함

```
1. 서버로부터 HTML 전달받음
2. HTML 파싱 → DOM 트리 생성 (메모리)
3. CSS 파싱 → CSSOM 트리 생성 (메모리)
4. DOM + CSSOM → Render 트리 (그릴 것만)
5. Layout → 위치/크기 계산
6. Paint → 색상/배경 채우기
7. Composite → 레이어 합성 → 화면 출력
```

## 3. 이벤트 루프

**Q. 이 코드 결과가 뭘까?**
```js
console.log('1')
setTimeout(() => console.log('2'), 0)
console.log('3')
```

0ms인데 결과는 `1 → 3 → 2`야.

---

**Q. 우와 저 0이라는게 헷갈리네 ㅋㅋㅋ 뭐야 달라?**

달라! 브라우저는 JS를 실행할 때 두 줄이 있어:

```
콜스택 (Call Stack)  →  지금 당장 할 일
큐 (Queue)          →  나중에 할 일
```

`setTimeout`은 콜스택에서 빼서 큐에 넣어. 0ms여도.

---

**Q. 너 잠깐만 멈춰봐~ 뭐 이런 느낌인데...**

ㅋㅋㅋㅋ 정확해!! `setTimeout`을 만나면 "잠깐, 이건 나중에 해" 하고 옆으로 빼놓는 거야.

---

**Q. 콜스택이 뭐야?**

스택(Stack) = 접시 쌓기. **LIFO — Last In, First Out**.

```
호출 → push (접시 올라감)
실행 → 접시 위에서 실행 중
끝   → pop  (접시 내려감)
```

```js
function a() { b() }
function b() { console.log('b!') }
a()
```

```
a() 호출  →  [a]
b() 호출  →  [b][a]
log 호출  →  [log][b][a]
log 끝    →  [b][a]
b 끝      →  [a]
a 끝      →  []
```

선언 순서가 아니라 호출 순서로 쌓여. 함수의 끝(`}` 또는 `return`)에서 pop.

---

**Q. 그럼 스택이라는게 단순히 호출 및 실행만 쌓는게 아니라 호출 → 실행 → 끝 이런 순서인거야?**

맞아!

```
호출   → 콜스택에 올라감  (push)
실행   → 콜스택 위에서 실행 중
끝     → 콜스택에서 내려감 (pop)
```

콜스택 = "지금 이 순간 뭐가 실행되고 있는가"의 스냅샷.

그래서 JS가 싱글 스레드라는 말이 나오는 거야 — 콜스택이 하나야. 한 번에 하나씩만 실행 가능.

---

**Q. 큐는?**

대기줄. **FIFO — First In, First Out**. 먼저 온 것이 먼저 나감.

```
콜스택  →  새치기 합법 (LIFO)
큐      →  새치기 불법 (FIFO)
```

---

**Q. 이벤트 루프가 하는 일은?**

딱 하나:
```
"콜스택 비었어? 그럼 큐에서 꺼내서 콜스택에 올려!"
```

쉬지 않고 계속 감시. 루프라는 이름이 붙은 이유.

요리 비유:
```
물을 끓인다     →  큐에 넣음 "다 됐으면 알려줘"
양념을 만든다   →  콜스택에서 계속 처리
물이 끓었으면   →  이벤트 루프가 큐에서 꺼냄
재료를 넣는다   →  실행
```

---

**Q. 요리 과정이 콜스택이며 콜스택은 계속해서 돌고 있고 그 중에 특정 스택은 큐에 넣을 수 있고 그걸 감시하는 이벤트 루프가 특정 상황이 되면 큐에서 가져와서 콜스택에 넣고 실행**

완벽해!! 그게 이벤트 루프 전체야.

```
콜스택      = 지금 하는 요리 과정
큐          = "다 되면 알려줘" 대기열
이벤트 루프 = 콜스택 비면 큐에서 꺼내주는 감시자
```

**Q. js는 싱글 스레드라고 했는데 다른 프로그래밍 언어에 멀티 스레드가 있어?**

응 있어! Java, C++, Python 등은 멀티 스레드 지원.

근데 UI는 어떤 언어든 싱글 스레드가 업계 표준이야:
```
Java  (Android) →  UI 스레드 하나
Swift (iOS)     →  Main 스레드 하나
JS    (Browser) →  메인 스레드 하나
```

DOM을 멀티 스레드로 다루면 동시에 같은 요소를 수정하는 충돌이 생겨서 의도적으로 싱글 스레드로 만든 거야.

**Q. 큐가 두 종류라고?**

```
마이크로태스크 큐  →  VIP 줄  (Promise.then, async/await)
매크로태스크 큐    →  일반 줄 (setTimeout, setInterval, 이벤트 핸들러)
```

우선순위:
```
콜스택 비워짐
→ 마이크로태스크 큐 전부 비움  ← VIP 먼저!
→ 매크로태스크 큐에서 하나 꺼냄
```

```js
setTimeout(() => console.log('매크로'), 0)
Promise.resolve().then(() => console.log('마이크로'))
console.log('콜스택')

// 결과: 콜스택 → 마이크로 → 매크로
```

맞아!! 데이터 통신 결과를 최대한 빨리 화면에 반영해야 하니까 VIP 대우를 받는 거야.

**Q. promise나 async/await은 다른 이벤트로도 사용하기도 할 수 있을 거 아냐?**

맞아! 데이터 통신만이 아니야. "기다려야 하는 것"이면 전부 쓸 수 있어:

```js
await new Promise(resolve => setTimeout(resolve, 1000))  // 타이머
await readFile('./data.json')                            // 파일 읽기
await Promise.all([fetch('/api/user'), fetch('/api/posts')]) // 동시 요청
```

**Q. 아하 비동기!!! 굳이 비동기 통신이 아니더라도 비동기 렌더링이라거나 비동기 연산이라거나 그럴 수도 있다는거네? 싱글 스레드이기 때문에?**

맞아!!! 완벽하게 연결됐다!!

```
싱글 스레드
→ 기다리면 전체가 멈춤
→ 기다림이 있는 건 전부 비동기로 처리
→ Promise/async/await으로 다룸

비동기 통신    →  서버 응답 기다림
비동기 렌더링  →  무거운 렌더링 기다림
비동기 연산    →  오래 걸리는 계산 기다림
```

React `Suspense`도 여기서 출발해. skeleton이랑도 연결되고 ㅋㅋ

---

**Q. 좋아 그럼 js도 멀티 스레드 비슷하게 쓴다는 거는 멀티 스레드 흉내를 위해서 비동기 통신을 한다는 거겠지**

비동기는 "기다리는 동안 다른 일 하는 것"이고, 멀티 스레드 흉내는 따로 있어 — **Web Worker**.

```
Web Worker = JS에서 진짜로 별도 스레드 생성
메인 스레드  →  UI 담당
Web Worker  →  무거운 연산 담당 (DOM 못 건드림)
```

우리 프로젝트 Sandpack도 Web Worker 사용 — 코드 실행을 별도 스레드에서 돌려서 UI가 버벅이지 않아.

```
1. 주소 입력
2. DNS 조회 → IP 주소 획득
3. 해당 IP 서버에 요청
4. 서버가 HTML 반환
5. 브라우저 렌더링
```

---

**Q. DNS가 뭐야?**

도메인을 IP 주소로 변환하는 전화번호부.

```
codedojang.vercel.app  →  76.76.21.21
```

캐싱 레벨: 브라우저 → OS → 라우터 → DNS 서버. 가까운 곳에서 찾으면 거기서 끝.

---

**Q. 캐시... cache의 근원적인 뜻이 궁금해지네 ㅋ**

프랑스어 **cacher** — "숨기다". 나중에 쓰려고 몰래 숨겨둔 것.

```
다람쥐가 겨울 대비해서 도토리를 땅에 숨겨둠 = 캐시
```

발음도 재밌어 — "캐시"인데 cash(현금)랑 똑같이 발음해. 영어권 개발자들도 헷갈려 ㅋㅋ

---

**Q. HTTP가 뭐야?**

**HyperText Transfer Protocol**

```
Hyper     →  단순 텍스트를 넘어선
Text      →  텍스트 기반
Transfer  →  전송
Protocol  →  규약, 약속
```

---

**Q. 상태코드 정리해줘**

```
2xx  →  성공    (200 성공, 201 생성됨, 204 내용없음)
3xx  →  이동    (301 영구이동, 302 임시이동, 304 캐시그대로)
4xx  →  내 잘못 (400 잘못된요청, 401 인증필요, 403 권한없음, 404 없음, 429 너무많음)
5xx  →  서버 잘못 (500 서버에러, 502 게이트웨이에러, 503 서버점검중)
```

401 vs 403:
```
401  →  "너 누구야?" (로그인 안 됨)
403  →  "알긴 아는데 안 돼" (권한 없음)
```

보안 팁 — 403 대신 404 반환하면 API 존재 자체를 숨길 수 있음.

---

**Q. HTTP 캐시는 어떻게 동작해?**

서버가 응답할 때 헤더에 지시를 같이 보냄:

```
Cache-Control: max-age=3600  ← "1시간 동안 캐시 써도 됨"
```

속도 순서:
```
캐시 유효기간 내  →  서버 요청 없이 바로 씀  (제일 빠름)
캐시 만료 후     →  304로 확인
  안 바뀜        →  304 → 캐시 재사용
  바뀜           →  200 + 새 파일          (제일 느림)
```

---

**Q. Cache-Control 옵션들은?**

```
max-age=3600  →  유효기간
no-cache      →  저장은 하되 매번 서버 확인 (no-cache인데 저장함 — 헷갈리는 이름 ㅋ)
no-store      →  저장 자체 금지 (인터넷뱅킹, 결제정보)
public        →  CDN 포함 어디서든 캐시 가능
private       →  브라우저만 캐시 가능
```

---

**Q. 저 정책은 프론트에서 정하는 거야 아니면 백엔드에서 정하는 거야?**

Cache-Control 헤더는 서버(백엔드)가 응답에 담아서 보냄. 우리 프로젝트는 Next.js + Vercel이라 백엔드 없이 Next.js가 자동 설정.

---

**Q. fetch 말고 다른 호출 방식이 있어?**

```
브라우저 내장:
  fetch           →  현재 표준
  XMLHttpRequest  →  옛날 방식 (jQuery $.ajax)

라이브러리:
  axios           →  fetch 불편한 거 해결 + 인터셉터
  react-query     →  데이터 가져오기 + 캐싱 + 로딩/에러 상태
  swr             →  Vercel이 만든 react-query 경쟁자
```

---

**Q. 데이터를 호출한다 라는 레이어를 넣는다면 어디가 좋아?**

레이어 3(UI 설계)과 레이어 4(성능) 사이. UI 설계 끝나면 "이 UI에 데이터를 어떻게 연결하지?" 가 자연스러운 다음 질문이거든.

→ **레이어 4 — 데이터를 다룬다** 로 독립, 기존 4~7은 5~8로 밀림.

## 5. ETag

**Q. ETag가 뭐야?**

파일의 지문(fingerprint).

```
파일 내용이 바뀌면  →  ETag도 바뀜
파일 내용이 같으면  →  ETag도 같음
```

```
서버 응답: 파일 + ETag: "abc123"

캐시 만료 후:
브라우저  →  "내가 가진 건 ETag: abc123 이야"
서버      →  같으면 304, 다르면 200 + 새 파일
```

## 6. 추상화 / 캡슐화 / 다형성

**Q. "React가 뭘 추상화하고 있다"는 게 무슨 말이야?**

추상화(抽象化):
```
抽  →  뽑을 추 (끌어내다)
象  →  모양 상
→  형상에서 본질을 뽑아내다
```

라틴어 abstrahere — abs(~로부터) + trahere(끌어당기다).

```
자동차 핸들  →  내부 몰라도 "돌리면 방향 바뀐다"만 알면 됨
useState    →  내부 몰라도 "값 바꾸면 화면 바뀐다"만 알면 됨
```

아래가 안 보이면 "왜 리렌더링이 자꾸 일어나지?" 를 모름. 아래가 보이면 원인을 찾을 수 있음.

---

**Q. 추상...이라는 말의 느낌을 이 나이 먹고 알게 되다니.... 抽象에서 추가 무슨 추야?**

抽 — 뽑을 추. 抽籤(추첨), 抽出(추출), 抽象(추상) 전부 "뽑아내다" 느낌.

象도 재밌어 — 원래 코끼리 상이야. 코끼리가 워낙 인상적인 동물이라 "인상", "형상", "모양"으로 의미 확장됨.

---

**Q. 캡슐화는?**

```
캡슐처럼 내부를 감싸서 보호
밖에서는 인터페이스로만 소통 가능
```

추상화 vs 캡슐화:
```
추상화  →  "뭘 하는지" 보여주는 것  (사용자 관점)
캡슐화  →  "어떻게 하는지" 숨기는 것 (설계자 관점)
```

import한 컴포넌트는 이미 둘 다 충족:
```js
import { Button } from '@/components/ui/button'
// 내부 구현 몰라도 됨 → 추상화 ✅
// 내부를 건드릴 수 없음 → 캡슐화 ✅
```

---

**Q. 다형성은?**

多形性 — 여러(多) 모양(形)을 가지는 성질(性).

```
소.울음소리()     →  "음메"
개.울음소리()     →  "멍멍"
고양이.울음소리() →  "야옹"
```

React 버전:
```jsx
<Button variant="primary" />   →  파란 버튼
<Button variant="danger" />    →  빨간 버튼
<Button variant="ghost" />     →  투명 버튼
```

의도적으로 설계된 "같은 이름, 다른 동작" — 실수로 생긴 카오스와 다름 ㅋㅋ
