## 1부 — URL 입력부터 렌더링까지

**Q. 브라우저 주소창에 URL 입력하면 어떤 일이 일어나?**

```
1. 주소 입력
2. DNS 조회 → IP 주소 획득
3. 해당 IP 서버에 요청
4. 서버가 HTML 반환
5. 브라우저 렌더링 (파이프라인)
```

## 2부 — DNS

**Q. DNS가 뭐야?**

도메인을 IP 주소로 변환하는 전화번호부야.

```
도메인  →  사람이 읽기 좋은 주소
IP      →  컴퓨터가 찾아갈 수 있는 실제 주소

codedojang.vercel.app  →  76.76.21.21
"코드도보통지"         →  010-xxxx-xxxx  (전화번호부와 같은 원리)
```

---

**Q. DNS 캐싱은?**

매번 DNS 조회하면 느리니까 저장해둬.

```
첫 번째 접속  →  DNS 조회 → IP 저장
두 번째 접속  →  저장된 IP 바로 사용 → DNS 조회 생략
```

캐싱 레벨:
```
브라우저 캐시  →  크롬이 기억
OS 캐시       →  윈도우/맥이 기억
라우터 캐시    →  공유기가 기억
DNS 서버 캐시 →  통신사가 기억
```

가까운 곳에서 찾으면 거기서 끝, 없으면 다음 레벨로.

도메인 바꿨는데 예전 사이트가 뜨는 이유 — DNS 캐시가 남아있는 것. 해결: flush DNS 또는 강력 새로고침.

## 3부 — HTTP

**Q. HTTP가 뭐야?**

**HyperText Transfer Protocol**

```
Hyper     →  단순 텍스트를 넘어선 (링크, 이미지 등 포함)
Text      →  텍스트 기반
Transfer  →  전송
Protocol  →  규약, 약속 (서로 간의 통신 방식)
```

편지 규약처럼 — 정해진 형식을 지켜야 배달이 됨.

기본 구조:
```
요청 (Request):
  "GET /training/01-useState HTTP/1.1"
  어떤 방식으로 / 어떤 주소를 / 어떤 버전으로

응답 (Response):
  "HTTP/1.1 200 OK"
  버전 / 상태코드 / 상태메시지
```

---

**Q. 상태코드 정리해줘**

숫자 앞자리로 분류:
```
1xx  →  정보 (거의 볼 일 없음)
2xx  →  성공
3xx  →  리다이렉트
4xx  →  클라이언트 잘못
5xx  →  서버 잘못
```

자주 보는 것들:
```
200  →  성공
201  →  성공 + 새로 생성됨 (POST 후)
204  →  성공 + 돌려줄 내용 없음 (삭제 후)

301  →  영구 이동 (주소 바뀜)
302  →  임시 이동
304  →  캐시 그대로 써도 됨

400  →  요청이 잘못됨
401  →  인증 필요 (로그인 안 됨) → "너 누구야?"
403  →  권한 없음 → "알긴 아는데 안 돼"
404  →  못 찾음
429  →  요청 너무 많음 (API 제한)

500  →  서버 에러
502  →  게이트웨이 에러 (서버 연결 실패)
503  →  서버 일시적 불능 (점검 중)
```

401 vs 403:
```
401  →  로그인 자체가 안 된 상태
403  →  로그인은 됐는데 권한 없음
        (내 계정으로 접근 불가한 API 호출 시)
```

보안 팁:
```
"이 API 있긴 한데 넌 권한 없어" → 403
→ 해커 입장에서 API 존재 확인됨

그래서 일부러 → 404
→ "그런 거 없어"로 숨겨버림
```

## 4부 — HTTP 캐시

**Q. 캐시(Cache)의 어원은?**

프랑스어 **cacher** — "숨기다"
```
나중에 쓰려고 몰래 숨겨둔 것
= 다람쥐가 겨울 대비해서 도토리를 땅에 숨겨두는 것
```

멀리서 가져오면 느리니까 가까운 곳에 숨겨두고 빠르게 꺼내 씀.

---

**Q. DNS 캐시랑 HTTP 캐시가 달라?**

같은 캐시 개념인데 대상이 달라:
```
DNS 캐시   →  IP 주소를 숨겨둠
HTTP 캐시  →  파일(HTML, CSS, JS, 이미지)을 숨겨둠
```

---

**Q. HTTP 캐시는 어떻게 동작해?**

서버가 응답할 때 헤더에 지시를 같이 보내:
```
서버 응답:
  파일 내용
  + Cache-Control: max-age=3600  ← "1시간 동안 캐시 써도 됨"
```

속도 순서:
```
캐시 유효기간 내   →  서버 요청 없이 바로 씀  (제일 빠름)
캐시 만료 후      →  304로 확인 요청
  └ 안 바뀜       →  304 Not Modified → 캐시 재사용
  └ 바뀜          →  200 + 새 파일    (제일 느림)
```

---

**Q. Cache-Control 옵션들은?**

```
max-age=3600   →  3600초(1시간) 동안 캐시 유효
no-cache       →  저장은 해, 근데 쓰기 전에 서버 확인
no-store       →  저장 자체를 하지 마 (흔적도 남기지 마)
public         →  CDN 포함 어디서든 캐시 가능
private        →  브라우저만 캐시 가능 (CDN 저장 안 됨)
```

no-cache vs no-store:
```
no-cache  →  저장은 하되 매번 확인
no-store  →  저장 자체 금지 (인터넷뱅킹 잔액, 결제 정보 등)
```

private vs public:
```
private  →  내 브라우저에만 (로그인한 개인 정보)
public   →  CDN에도 저장 가능 (누구나 같은 내용)
```

---

**Q. Cache-Control은 프론트가 정해? 백엔드가 정해?**

```
Cache-Control 헤더  →  서버(백엔드)가 응답에 담아서 보냄
```

현실:
```
보안 민감한 데이터  →  기획/보안팀에서 결정 → no-store
일반 정적 파일     →  관례대로 → max-age 길게
API 응답          →  백엔드 개발자가 결정
```

우리 프로젝트:
```
Next.js + Vercel → 백엔드 없음
→ Next.js가 Cache-Control 자동 설정
→ SSG 페이지는 자동으로 public, max-age 길게
```

## 5부 — 데이터 호출 방식

**Q. fetch 말고 다른 호출 방식이 있어?**

```
브라우저 내장 (설치 불필요):
  fetch           →  현재 표준
  XMLHttpRequest  →  옛날 방식 (jQuery $.ajax가 이걸 씀)

라이브러리 (설치 필요):
  axios           →  fetch 불편한 거 해결한 래퍼
  react-query     →  캐싱, 로딩, 에러 상태 관리까지
  swr             →  Vercel이 만든 react-query 경쟁자
```

axios가 아직도 많이 쓰이는 이유:
```
fetch의 불편한 점:
  에러 처리 불편 (404도 에러로 안 잡힘)
  요청/응답 인터셉터 없음
  JSON 자동 변환 없음

axios:
  위 단점 전부 해결
  인터셉터로 토큰 자동 주입 가능 ← 실무에서 자주 씀
```

react-query/swr은 한 단계 위:
```
fetch/axios   →  "데이터 가져오기"
react-query   →  "데이터 가져오기 + 캐싱 + 로딩상태 + 에러상태 + 재시도"
```

fetch 캐시 옵션:
```js
// 매번 새로 받아와야 할 때
fetch('/api/balance', { cache: 'no-store' })

// 서버에 확인 후 사용
fetch('/api/posts', { cache: 'no-cache' })

// Next.js SSG
fetch('/api/data', { cache: 'force-cache' })

// Next.js ISR — 1시간마다 갱신
fetch('/api/data', { next: { revalidate: 3600 } })
```

## 6부 — ETag

**Q. ETag가 뭐야?**

파일의 지문(fingerprint)이야.

```
파일 내용이 바뀌면  →  ETag도 바뀜
파일 내용이 같으면  →  ETag도 같음
```

동작 방식:
```
서버 응답:
  파일 내용
  + ETag: "abc123"   ← 이 파일의 지문

캐시 만료 후 브라우저 요청:
  "이 파일 아직 유효해? 내가 가진 건 ETag: abc123"

서버 확인:
  ETag 같으면  →  304 Not Modified (그대로 써)
  ETag 다르면  →  200 + 새 파일
```

max-age + ETag 조합:
```
max-age  →  유효기간 동안 서버 요청 자체를 안 함
ETag     →  유효기간 끝난 후 바뀌었는지 확인하는 수단
```

## 7부 — 프로그래밍 원칙 (대화 중 나온 것)

**Q. "React가 뭘 추상화하고 있다"는 게 무슨 말이야?**

추상화(抽象化):
```
抽  →  뽑을 추 (끌어내다)
象  →  모양 상 (형상)
→  형상에서 본질을 뽑아내다
```

라틴어 abstrahere — abs(~로부터) + trahere(끌어당기다)

```
복잡한 내부를 숨기고 단순한 인터페이스만 노출

자동차 핸들  →  "돌리면 방향 바뀐다"만 알면 됨
              조향축, 타이어 연결 몰라도 됨

useState    →  "값 바꾸면 화면 바뀐다"만 알면 됨
              Virtual DOM, Reconciler 몰라도 됨
```

---

**Q. 캡슐화는?**

```
캡슐(capsule) →  약 캡슐처럼 내부를 감싸서 숨김
               밖에서는 인터페이스로만 소통 가능
```

추상화 vs 캡슐화:
```
추상화  →  "뭘 하는지" 보여주는 것  (사용자 관점)
캡슐화  →  "어떻게 하는지" 숨기는 것 (설계자 관점)
```

컴포넌트로 보면:
```jsx
<Button variant="primary" onClick={handleClick}>저장</Button>

추상화  →  "버튼이다, 누르면 뭔가 된다"만 알면 됨
캡슐화  →  Button 내부 상태를 밖에서 직접 못 건드림
           props로만 소통 가능
```

import한 컴포넌트는 추상화 + 캡슐화가 이미 완성된 것:
```js
import { Button } from '@/components/ui/button'
// 내부 구현 몰라도 됨 → 추상화 ✅
// 내부를 건드릴 수 없음 → 캡슐화 ✅
```

---

**Q. 다형성은?**

```
多形性  →  여러(多) 모양(形)을 가지는 성질(性)
```

```js
소.울음소리()     →  "음메"
개.울음소리()     →  "멍멍"
고양이.울음소리() →  "야옹"
```

같은 이름, 누가 부르냐에 따라 다른 동작.

React 버전:
```jsx
<Button variant="primary" />    →  파란 버튼
<Button variant="danger" />     →  빨간 버튼
<Button variant="ghost" />      →  투명 버튼
```

## 핵심 인사이트

1. **URL 입력 → DNS → IP → HTTP 요청 → 렌더링** — 브라우저가 페이지를 여는 전체 흐름
2. **캐시 = 숨겨둔 것** — 멀리서 가져오면 느리니까 가까운 곳에 저장
3. **304 = 파일 전송 없이 "그대로 써" 신호** — ETag로 변경 여부 확인
4. **Cache-Control은 백엔드가 설정** — 프론트는 이를 이해하고 활용
5. **추상화 + 캡슐화 = 잘 만든 컴포넌트** — import해서 쓰는 순간 이미 적용된 것
6. **다형성 = 같은 이름, 다른 동작** — 의도적 설계 (카오스와 다름)
